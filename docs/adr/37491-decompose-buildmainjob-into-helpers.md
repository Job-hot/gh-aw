# ADR-37491: Decompose buildMainJob into single-responsibility helpers

**Date**: 2026-06-07
**Status**: Draft

## Context

Daily `largefunc` lint tracking flagged function-length violations concentrated in `pkg/workflow`. In `compiler_main_job.go`, `buildMainJob` had grown into a long, multi-concern method that assembled the agent job end-to-end: step-list construction (setup/checkout, runtime paths, main steps), job-condition derivation (activation handoff plus the daily-effective-tokens guardrail), dependency resolution (base needs plus custom jobs referenced in markdown/custom-steps/`engine.env`), built-in `needs.*` warning emission, output assembly (model/token metrics, safe-output and PR-checkout outputs, engine error-detection outputs), env assembly (safe-output and cache-memory variables), and permission synthesis (contents-read augmentation, gh-write enforcement, inferred read scopes). The method was a cyclomatic-complexity hotspot whose branches were hard to read or test in isolation, while its generated-YAML output contract had to be preserved exactly.

## Decision

We decided to decompose `buildMainJob` into a thin orchestrator that delegates to focused, single-responsibility helpers, preserving existing output and behavior exactly. `buildMainJob` now calls `buildMainJobStepList`, `buildMainJobCondition`, `buildMainJobDependencies` (which composes `getMainJobBaseDependencies` and `buildEngineEnvDependencyContent`), `warnOnBuiltInEngineEnvNeeds`, `buildMainJobOutputs`, `buildMainJobEnv`, and `buildMainJobPermissions`. Permission shaping is further split into the pure helpers `ensureMainJobContentsRead` and `extractMainJobScripts`. This follows the same plain-helper-function convention established by [ADR-36694](36694-decompose-buildcustomjobs-into-helpers.md) and [ADR-37339](37339-decompose-activation-job-builder-into-helpers.md).

## Alternatives Considered

### Alternative 1: Leave the method as-is (suppress the lint rule)
We could have left the monolithic `buildMainJob` in place and suppressed the `largefunc` warning with an inline directive or per-file exclusion. This avoids churn and regression risk, but does nothing to reduce cyclomatic complexity, keeps the multi-concern branches hard to test, and normalizes silencing the lint signal the daily tracking exists to surface. Rejected because it defers the maintainability problem indefinitely.

### Alternative 2: Introduce a dedicated `MainJobBuilder` type
We could have extracted the logic into a builder struct holding `c`, `data`, and `activationJobCreated`, exposing fluent methods like `withSteps()` and `withPermissions()`. This would centralize state, but introduces a new type and lifecycle, diverges from the free-function/method style used throughout the compiler, and is a larger structural change than a lint-driven decomposition warrants. Rejected in favor of plain helpers that match the precedent set by ADR-36694 and ADR-37339.

## Consequences

### Positive
- The `buildMainJob` orchestrator drops below the `largefunc` threshold, making the high-level agent-job assembly flow readable at a glance.
- Individual concerns (condition derivation, dependency scanning, permission synthesis) can now be reasoned about and tested independently â€” e.g. `ensureMainJobContentsRead` gains targeted unit coverage in `compiler_main_job_test.go`.
- Pure helpers like `ensureMainJobContentsRead` and `extractMainJobScripts` remove inline branching and make the permission logic reusable and self-documenting.

### Negative
- The number of functions in `compiler_main_job.go` increases, so navigation now requires jumping between several small functions rather than scrolling one method.
- This behavior-preserving refactor carries regression risk in subtle output ordering and conditional emission (e.g. the order in which `outputs`/`env` map entries and `depends` are appended, and the warning dedup using a set rather than a `map[string]bool`); this relies on existing snapshot/compiler tests to catch divergence.

### Neutral
- Generated workflow YAML output shape is unchanged; this is purely an internal restructuring with no user-visible or schema impact.
- The built-in-job warning dedup switched from `map[string]bool` to a `map[string]struct{}` set, with no change in which warnings are emitted.
