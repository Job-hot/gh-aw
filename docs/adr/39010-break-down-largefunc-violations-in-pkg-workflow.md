# ADR-39010: Break Down Oversized Functions in `pkg/workflow` via Extract-Method

**Date**: 2026-06-13
**Status**: Draft

## Context

The `pkg/workflow` package carries a backlog of functions that exceed the project's 60-body-line `largefunc` lint limit (273 violations at the time of this PR). Several of the largest offenders — workflow step generators in `compiler_yaml_main_job.go` and the Copilot tool-argument builder in `copilot_engine_tools.go` — had grown to 100–208 body lines, mixing many distinct concerns in a single scope. These large functions are hard to read, hard to test in isolation, and repeatedly trip the linter that enforces the existing function-size policy. This PR reduces the violation count by 10 (273 → 263) without changing behaviour.

## Decision

We will reduce oversized functions in `pkg/workflow` by applying extract-method refactoring: each function over the limit is decomposed into focused, package-private helper functions, each itself ≤60 body lines. The decomposition preserves observable behaviour (the regenerated `.lock.yml` fixtures are unchanged in intent) and follows the existing structure of the package rather than introducing new abstractions, types, or public API.

## Alternatives Considered

### Alternative 1: Raise or disable the `largefunc` threshold
We could loosen the lint limit (or suppress it per-function with `//nolint`) instead of refactoring. Rejected because the function-size limit is an intentional, codebase-wide quality gate (see the custom-linter ADR-38742); weakening it for convenience would erode the policy and hide genuinely unwieldy functions.

### Alternative 2: Extract logic into new types or sub-packages
The largest functions could be moved behind new struct types or a dedicated sub-package. Rejected as over-engineering for a lint-conformance pass: it would expand the public surface, churn imports, and introduce architectural change where a behaviour-preserving local refactor suffices.

## Consequences

### Positive
- The 10 refactored functions now satisfy the `largefunc` lint limit, shrinking the violation backlog.
- Smaller, named helpers improve readability and make individual step-generation concerns independently testable.

### Negative
- The number of package-private functions and call-site indirection increases, adding navigation hops when tracing a single flow.
- Extract-method on hot generation paths risks subtle behaviour drift if a helper's inputs/outputs are mis-threaded; correctness rests on the unchanged `.lock.yml` fixtures.

### Neutral
- Many `.github/workflows/*.lock.yml` files are regenerated as part of the change; the diff line count is inflated by these fixtures rather than by new business logic.
- No public API, type, or behaviour changes; the refactor is internal to `pkg/workflow`.
