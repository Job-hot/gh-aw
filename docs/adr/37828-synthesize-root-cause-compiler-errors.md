# ADR-37828: Surface root-cause-first compiler errors via post-hoc synthesis and headline-based classification

**Date**: 2026-06-08
**Status**: Draft

## Context

When a workflow frontmatter has an invalid `engine:` value (e.g. `engine: claud`), schema validation produced a cascade of low-value messages (`oneOf` failures, unrelated `Unknown property` fallout) that buried the actual root cause. Error classification in `pkg/workflow/error_recovery.go` matched substrings across the *entire* message body, so embedded YAML context lines (e.g. a line containing `pull_request`) could trigger an unrelated suggestion ("Correct the event or filter name…"). Separately, the frontmatter context extractor's regex did not match `yaml.FormatError` lines prefixed with `>`, so the highlighted error line was sometimes dropped from the rendered snippet. These three defects degraded the quality of the first actionable fix presented to users running `gh aw compile`.

## Decision

We will improve compiler error quality through three coordinated changes in the error-recovery layer rather than at the schema-validation source:

1. **Synthesize a dedicated root-cause error for invalid engine typos.** In `ExpandErrorMessages`, a new `synthesizeInvalidEngineTypoMessage` uses a regex to detect an `engine: <unknown>` context line, validates the value against `constants.AgenticEngines`, and emits a critical `unknown engine "..."` message (with the valid engine list and a nearest-match suggestion via `parser.FindClosestMatches`) ahead of the downstream schema noise.
2. **Classify by the message headline, not the full body.** `classifyErrorMessage` now extracts the first line (`headline`) and matches categories against it, preventing embedded context lines from producing spurious suggestions. A new `missing ':' after key` case yields a targeted syntax suggestion, and `unknown engine` joins the existing engine-configuration case.
3. **Preserve highlighted error lines in context.** `sourceContextPattern` in `frontmatter_error.go` is widened to match optional `>`-prefixed line markers so the reported line is retained in the snippet.

## Alternatives Considered

### Alternative 1: Emit the engine-typo error at the schema-validation source
The unknown-engine diagnostic could be produced where engine validation runs, instead of reverse-engineering it from formatted error text with a regex in the recovery layer. This was not chosen (for now) because the schema layer reports a generic `oneOf` failure with no awareness of the typo, and threading typo-aware messaging through the validator is a larger change than post-hoc synthesis; the recovery layer already owns error prioritization and de-duplication, making it the lowest-risk insertion point. *[TODO: verify whether enriching the validator was evaluated and explicitly deferred.]*

### Alternative 2: Keep full-body classification but exclude context lines
Rather than restricting classification to the headline, the matcher could continue scanning the whole message while filtering out lines that look like YAML context (`N | ...`). This was not chosen because maintaining an exclusion list of context-line shapes is more fragile than simply keying off the headline, which is the authoritative summary of the error.

## Consequences

### Positive
- Invalid engine values now surface an explicit, actionable root cause ("unknown engine "claud". … Did you mean "claude"?") as the first message, ahead of schema noise.
- Headline-based classification eliminates accidental event/filter suggestions triggered by embedded YAML context lines, improving suggestion accuracy.
- The rendered error context reliably includes the exact reported line (including the `>`-highlighted line), guarded by a regression test.

### Negative
- `synthesizeInvalidEngineTypoMessage` parses formatted error text with regexes (`engineContextLinePattern`, `errorFilePathPattern`) rather than structured error data, so it is brittle to changes in how validation errors and YAML context are formatted.
- Engine-name validity is now checked in the error-recovery layer in addition to the schema layer, duplicating knowledge of `constants.AgenticEngines` and the "valid engine" invariant across two boundaries.

### Neutral
- The synthesized message reuses the `path:line:col: error:` format and is de-duplicated through the existing `seen` set in `ExpandErrorMessages`.
- Regression tests were added for the synthesized engine-typo message, the missing-colon syntax suggestion, and the presence of the exact reported YAML error line in rendered context.
- Engine names are presented sorted (`sortedAgenticEngines`) for stable, deterministic output.
