# ADR-37672: Migrate Legacy max-effective-tokens Budgets to max-ai-credits

**Date**: 2026-06-07
**Status**: Draft

## Context

gh-aw previously expressed agent budgets using `max-effective-tokens` (ET), but budgeting has since moved to an AI-credit model (`max-ai-credits`) that the downstream AWF API proxy now consumes directly. During the transition both fields could coexist in frontmatter, in imported/shared workflow settings, and in enterprise default environment variables, leaving the compiler to emit two parallel budget concepts and forcing the proxy to understand a retired field. We need workflows that still carry the legacy ET field to keep working without manual edits, while removing the ambiguity that arises when a workflow specifies both budgets or uses an ET value that cannot be safely converted at compile time.

## Decision

We will treat `max-effective-tokens` as a deprecated, legacy-only input that the compiler automatically converts to `max-ai-credits` using a single shared ratio (`EffectiveTokensPerAICredit = 10000`, floor division), preserving `-1` as the disable sentinel. We will reject ambiguous legacy usage — specifying both `max-effective-tokens` and `max-ai-credits` in the same compiled workflow, or supplying an expression-valued `max-effective-tokens` that cannot be converted at compile time — by failing fast with a clear error. Generated AWF config will emit only `apiProxy.maxAiCredits` and will stop writing the retired `apiProxy.maxEffectiveTokens` field entirely, with token steering enablement following the AI-credit path only. The conversion ratio and helper logic are centralized (in `pkg/constants` and `pkg/typeutil`) so parser validation, compiler conversion, and AWF config generation share identical behavior.

## Alternatives Considered

### Alternative 1: Keep emitting both budget fields side by side
Continue writing `maxEffectiveTokens` alongside `maxAiCredits` in AWF config and let the proxy reconcile them. Rejected because it perpetuates two competing budget concepts, requires the proxy to keep understanding a retired field, and offers no single source of truth — the very ambiguity this change is meant to eliminate.

### Alternative 2: Hard-remove max-effective-tokens with no migration
Drop support for `max-effective-tokens` outright and require every workflow and enterprise default to switch to `max-ai-credits` manually. Rejected as an unnecessarily breaking change: existing workflows and shared imports would fail to compile until edited, whereas automatic conversion preserves backward compatibility for the common numeric case.

### Alternative 3: Convert at runtime instead of compile time
Leave the legacy field in generated config and convert ET to AI credits inside the runtime/proxy. Rejected because it pushes legacy-format knowledge downstream, delays validation of impossible-to-convert values (expressions, conflicting fields) to runtime, and keeps the retired field alive in generated artifacts.

## Consequences

### Positive
- Downstream AWF config and the API proxy deal with a single budget concept (`maxAiCredits`), removing the retired `maxEffectiveTokens` field from generated artifacts.
- Existing workflows, imports, and enterprise defaults that still use numeric `max-effective-tokens` continue to compile via automatic conversion — no manual migration required for the common case.
- Ambiguous or unconvertible configurations (both budgets set, or expression-valued legacy budgets) now fail fast at compile time with actionable error messages instead of producing surprising runtime behavior.
- A single centralized conversion ratio and helper guarantees parser validation, compiler conversion, and config generation stay consistent.

### Negative
- The fixed 10000-ET-per-credit ratio with floor division loses precision: values below 10000 ET (or remainders) convert down, and budgets under one credit are rejected rather than silently rounded.
- Expression-based `max-effective-tokens` values are no longer supported, which is a breaking change for any workflow that computed the legacy budget dynamically; such workflows must move to `max-ai-credits`.
- The legacy `MaxEffectiveTokens` field and parsing path still linger in the codebase for conversion purposes, so the deprecation is not yet a full removal.

### Neutral
- The conversion ratio (10000) is now a hard-coded constant in `pkg/constants`; changing the credit economics later requires updating that single constant.
- `-1` continues to act as the universal "disable budget / disable token steering" sentinel, mapped straight through to `max-ai-credits: -1`.
